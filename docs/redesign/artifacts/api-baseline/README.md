# API Baseline Capture (PHASE-0.1)

This directory contains a captured regression oracle of the **current, pre-redesign** backend
REST API behavior. It exists so later redesign phases (Phase 1 contract tests, Phase 8
frontend/backend parity checks) have a concrete, byte-level example of "what the API actually
does today" to diff against, instead of relying on the OpenAPI/spec docs alone.

## What's here

Every file was generated by one JUnit test,
`src/test/java/ua/kostenko/battleship/battleship/ApiBaselineOracleCaptureTest.java`, which drives
**one full two-player game** against a real, running Spring Boot instance
(`@SpringBootTest(webEnvironment = RANDOM_PORT)` + `TestRestTemplate`) on the **UKRAINIAN**
edition, calling every endpoint under `/api/v2/game/**` at least once, in the order a real
frontend session would: list editions, create a session, create two players, place all 10 ships
for both players (including one delete + re-add), start both players, fetch opponent info and
gameplay state, fire shots until one fleet is completely sunk (covering `MISS`, `HIT`, and
`DESTROYED` results, plus the final `hasWinner=true` state for both players), and finish with one
deliberately invalid call to capture a `400 ExceptionDto` sample.

For every HTTP call the test makes, three files are written here, named
`NN-<slug>.<kind>.json`, where `NN` is a zero-padded, strictly increasing step number (in call
order) and `<slug>` is a short kebab-case description of the call:

- `NN-<slug>.request.json` — pretty-printed request body (omitted for calls with no body, e.g.
  GETs and the `start` POST).
- `NN-<slug>.response.json` — pretty-printed response body.
- `NN-<slug>.meta.json` — `{"method": "...", "path": "...", "status": NNN}` for that call.

`manifest.json` is a JSON array of every step, in order, each entry shaped like
`{"slug": "...", "method": "...", "path": "...", "status": NNN}` — a compact index into the rest
of the directory.

There are 78 steps / 205 files in the current capture, covering every endpoint listed in the
PHASE-0.1 ticket:

- `GET /api/v2/game/editions`
- `POST /api/v2/game/sessions`
- `POST /api/v2/game/sessions/{sessionId}/players` (x2)
- `GET /api/v2/game/sessions/{sessionId}/state`
- `GET /api/v2/game/sessions/{sessionId}/changesTime`
- `GET /api/v2/game/sessions/{sessionId}/players/{playerId}/preparationState`
- `PUT /api/v2/game/sessions/{sessionId}/players/{playerId}/ships/{shipId}` (all 10 ships, both
  players, plus one re-add after delete)
- `DELETE /api/v2/game/sessions/{sessionId}/players/{playerId}/ships`
- `POST /api/v2/game/sessions/{sessionId}/players/{playerId}/start` (x2)
- `GET /api/v2/game/sessions/{sessionId}/players/{playerId}/opponent`
- `GET /api/v2/game/sessions/{sessionId}/players/{playerId}/state` (preparation + gameplay,
  including the final post-game state for both players)
- `POST /api/v2/game/sessions/{sessionId}/players/{playerId}/field/shot` (repeated; covers MISS,
  HIT, DESTROYED, and the invalid out-of-bounds 400 case)

## Regenerating

```
mvn test -Dtest=ApiBaselineOracleCaptureTest
```

This wipes and rewrites every file in this directory (the test creates the directory if missing
via `Files.createDirectories`, and each run's `NN-<slug>` filenames are deterministic given the
fixed ship layout and shot sequence hardcoded in the test, so a clean rerun produces the same set
of filenames with fresh IDs/timestamps inside). The test also carries real JUnit/AssertJ
assertions on status codes and key response fields, so it functions as a live regression check in
`mvn test`, not just a capture script.

## Non-deterministic fields — do NOT byte-compare these

Any future diffing tooling (Phase 1 contract tests, Phase 8 parity checks) must normalize or
ignore the following fields before comparing captures run-to-run or against a new implementation,
since they are expected to differ on every run even when behavior is otherwise identical:

- **`sessionId`, `playerId`, `shipId`** — server-generated UUIDs (see `IdGeneratorImpl`), fresh on
  every run. They also appear embedded inside the `path` field of every `*.meta.json` and inside
  `manifest.json`.
- **`changesTime` → `lastId`** (`06-get-changes-time.response.json`) — a timestamp-derived value
  from `GameControllerApi#getLastSessionChangeTime`, unique per run.
- Any other field that structurally embeds one of the above (e.g. `winnerPlayerName` is stable —
  it's the literal string `"Player_1"` — but nothing that echoes back a session/player/ship ID is
  stable).

Everything else (ship sizes, field layouts, `shotResult` values, `hasWinner`/`isPlayerWinner`
booleans, HTTP status codes, error message shapes) is deterministic given the fixed ship layout
and shot sequence the test uses, and safe to byte-compare after normalizing the IDs/timestamp
above.
